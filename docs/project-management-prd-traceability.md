# ProjectManagement 旧版 PRD 需求追踪矩阵

## 1. 目的、范围与证据口径

本矩阵服务于 Linear `HAO-428`，将已经定位到的权威旧版 PRD 逐项映射到当前 ProjectManagement 的需求基线、源码证据和 Linear 交付链。它是范围核对表，不以接口、实体或测试文件的存在替代真实验收。

### 权威需求来源

- **来源**：[BoQiang.ProjectManagerBS PRD V2.1（源码基线·全量研发周期版）](https://linear.app/haoyida/document/boqiangprojectmanagerbs-prd-v21源码基线全量研发周期版-a5de82f2967a)（Linear document `95c1aa5f-52f9-4ebe-b54d-94ccdf1b0ab3`，2026-07-17 更新）。
- **逐项范围**：该文档第 4.1–4.6 节的全部 35 个项目管理功能条目；第 6 节 M0–M7 作为交付周期约束单独追踪。
- **范围结论**：PRD 明确要求全量实现，不采用 MVP 或默认裁剪。任何标记为 `Missing` 的项目仍保留在路线图中，不代表删除或降级。经用户明确确认，项目管理范围内的真实身份浏览器 E2E/UAT 证据登记为允许例外，不得将其缺失表述为已通过；其余自动化测试、契约、权限和边界证据仍必须如实记录。

### 状态定义

| 状态 | 含义 |
|---|---|
| `Implemented（源码）` | 当前分支已有项目域源码、权限/数据边界和自动化测试证据；仍需以真实身份完成 API/页面验收。 |
| `Partial` | 已有实体、服务、控制器、页面或测试中的至少一部分，但缺少 PRD 所要求的完整交互、边界、集成或运行验收。 |
| `Missing` | 未定位到满足该功能的 ProjectManagement 生产链路；已映射的 Linear Case 是待交付追踪项。 |

## 4. 复用平台与新增项目域边界

| 类别 | 结论 | 可复用平台能力 | 必须由 ProjectManagement 承担的新增语义 |
|---|---|---|---|
| 身份、权限与隔离 | 复用基础设施，不复用为项目规则 | 认证、会话、`PermissionCode`、租户/应用上下文、ORM 过滤器 | 项目角色、Lead 范围、项目对象访问、导出/附件/批量同口径过滤。 |
| 文件、通知、IM、SignalR、Hangfire、审批 | 复用平台服务，不复制平台资源 | 对象存储/预览、通知投递、IM、实时通道、作业、工作流 | 项目/任务关联键、成员收敛、领域事件、幂等、权限回收和项目域失败补偿。 |
| 数据中心与审计 | 复用受控存储、追踪和平台运维能力 | 数据空间、SQLite 备份基础、TraceId、操作日志 | `.bqsync` 协议、项目同步日志、冲突策略、高风险维护锁、项目审计投影。 |
| 项目管理核心 | 必须新增原生项目域 | 无可替代的通用模块 | `Project`、`Milestone`、`Task`、依赖、进度、视图、回收站、业务命令及其专属权限/数据过滤。 |

## 5. 关闭规则与回溯方法

1. 每个 `PM-PRD-*` 的关闭必须回写对应 Linear Case，并附上构建、自动化测试、真实身份 API/UI、允许与拒绝路径、隔离和关键失败边界证据。
2. 只有同一功能的源码、API、前端、权限、ORM 数据过滤和数据持久化链路全部可追溯时，状态才可从 `Partial` 提升为 `Implemented（源码）`；再完成运行验收后才可关闭 Case。
3. 对 M6/M7 的 `Missing` 项不得以“没有 Case”删除或默认裁剪；应在 BoQiang.ProjectManager 项目下补充可验收的子 Case，并回填本矩阵。
4. 需求变更必须先更新 PRD 或建立经批准的版本记录（关联 `HAO-430`），然后更新本矩阵的需求 ID、Linear 映射和验收差距。
