1. **范围已纳入**：需求仍是路线图承诺，必须有追踪与阶段门。
2. **源码已定位**：仅说明存在静态入口、实体、接口、页面或测试证据。
3. **已实现并可关闭**：必须同时具备最新源码构建、自动化测试、真实身份 API/UI、允许/拒绝、隔离和失败边界证据。本基线不因前两者而把任何 `Partial`/`Missing` 能力升格为已实现。

## 5. 冻结、回退与范围变更入口

### 5.1 冻结准入

HAO-426 协调者在 Linear 留下批准记录时，必须同时写入 `PM-BL-20260718.1`、本文件提交 SHA、PRD document ID/`updatedAt`、第 2 节四项输入提交，以及批准人、时间和条件。批准前复核以下条件：

1. 第 3.1 节的 35 个 ID 连续且每项在追踪矩阵有 Linear 承接；
2. M1–M7 均显式保留，M6/M7 的“待拆分”不被当作删除；
3. 第 4 节风险已保留而非通过标记 `Implemented` 被掩盖；
4. HAO-430 的 DoR/DoD、阶段门和批准角色可执行。

冻结只固定范围与验收口径，不冻结实现分支，也不把任何 `Partial`/`Missing` 能力转为 Done。

### 5.2 变更入口

冻结后发生范围、验收、API/DTO/事件、数据/迁移、权限/隔离、架构/依赖、计划或紧急缺陷变化时，必须先在 Linear 建立唯一 `PM-CR-YYYYMMDD-NNN`，关联本基线、受影响 `PM-PRD-*` 与 HAO Case，并按治理文档完成影响矩阵与批准。仅内部重构、补测试或不改变需求/契约/验收的修复可在原 Case 留证；判断不确定时按需创建 `PM-CR-*`。

获批变更生成新基线 `PM-BL-*`：向后兼容澄清/补充递增 minor；范围、接口、数据或权限的审批级调整递增 major。新基线须标记 `Supersedes`，旧基线保持只读可追溯。

### 5.3 回退入口

回退是新建 `PM-CR-*` 的治理状态切换，不得使用 `git reset` 或改写 Linear 历史。变更单须说明撤回原因、目标基线、受影响源码/迁移/配置/数据/测试/发布 Case 和回归计划；独立实现 Case 负责实际代码/数据回退及验证。新的有效基线记录为 `Active`，被撤回基线标记 `Superseded/RolledBack`，并保留全部 SHA、批准和失败证据。

## 6. 使用与复核规则

1. 计划、设计、开发、测试和发布 Case 必须引用 `PM-BL-20260718.1` 与对应 `PM-PRD-*`；没有 ID 的需求先走变更入口。
2. 关闭任一 Case 前，按 `docs/project-management-requirements-governance.md` 的 DoD 填回 Git SHA、测试命令/结果、真实 API/UI 证据及未覆盖项；匿名 `401` 不构成权限验收。
3. 本文件是索引与冻结记录，不复制或覆盖能力矩阵、PRD 映射、词汇表或治理文档。若这些输入需要修改，先按第 5.2 节产生新基线，再更新其引用。
