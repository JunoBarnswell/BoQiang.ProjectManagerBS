# ProjectManagement 需求版本、范围变更与验收基线治理

## 1. 目标与适用范围

本文是 Linear `HAO-430` 的可执行治理规则。它约束 ProjectManagement 从 M0 到 M7 的需求版本、范围变更、验收基线和关闭证据；不引入未实现的审批、工单或自动化系统。

治理对象的固定关联如下：

| 对象 | 权威用途 | 当前定位 |
|---|---|---|
| PRD V2.1 | 产品范围的权威来源 | Linear document `95c1aa5f-52f9-4ebe-b54d-94ccdf1b0ab3`，不得以口头需求替代。 |
| 旧 PRD 追踪矩阵 | 原始需求到源码/Linear 的反向追踪 | `docs/project-management-prd-traceability.md`；包含 `PM-PRD-001` 至 `PM-PRD-035`。 |
| 能力与验收基线 | 当前源码能力与尚未闭环的缺口 | `docs/project-management-capability-matrix.md`。 |
| 产品与领域架构 | 聚合、权限、数据过滤、命令和运行边界 | `docs/project-management-product-architecture.md`。 |
| Linear | 任务、评审决策、执行和关闭证据 | `HAO-426` 是范围基线主题；`HAO-430` 是治理机制 Case；所有实现均须映射到 M0–M7 的主题/子 Case。 |
| Git | 文档/源码版本和不可变交付快照 | 以已合入提交 SHA 为准；不改写历史来“回滚”基线。 |

### 1.1 术语和编号

| 名称 | 格式 | 含义 |
|---|---|---|
| 需求 ID | `PM-PRD-NNN` | 来自追踪矩阵的稳定需求标识。新增需求从未使用的 `036` 起编号，并先写入追踪矩阵。 |
| 基线 ID | `PM-BL-YYYYMMDD.N` | 经批准且冻结的范围、验收、版本和证据集合，例如 `PM-BL-20260718.1`。 |
| 变更单 ID | `PM-CR-YYYYMMDD-NNN` | 每次影响已冻结或已进入开发需求的独立变更记录。 |
| 实现 Case | `HAO-NNN` | 一个可由单个责任人完成、验证和关闭的 Linear 子项；不能用主题父项代替最小交付单元。 |
| 基线版本 | `major.minor` | `major` 表示范围/接口/数据/权限的破坏性或审批级调整；`minor` 表示向后兼容的澄清、验收补充或新增未冻结项。 |

基线 ID、版本、PRD 文档修订时间、追踪矩阵 Git SHA、能力矩阵 Git SHA、架构文档 Git SHA 和关联 Linear URL 必须同时出现在基线记录中。没有这些字段的“当前版本”不是可用基线。

## 2. 基线建立、冻结和回退

### 2.1 建立与冻结步骤

基线管理员按下列顺序建立；当前仓库没有自动化编排器，因此每一步由责任人在 Linear 与 Git 中留下可审计链接或 SHA。

1. 选定 PRD V2.1 的具体修订，并列出纳入的 `PM-PRD-*`；不得以“所有功能”替代 ID 列表。
2. 在 Linear `HAO-426` 或本次范围基线的专用主题中建立/更新基线记录，写入 `PM-BL-*`、版本、M0–M7 覆盖范围、责任人和批准人。
3. 固定四份文档的 Git SHA：PRD 导出/引用说明、追踪矩阵、能力矩阵、产品架构。若 PRD 是 Linear 文档，则记录其 URL、document ID 和 `updatedAt`。
4. 对每个纳入项确认至少一个 Linear 主题和一个已存在或待创建的可验证子 Case；M6/M7 尚未细化的项目必须标记为“待拆分”，不能从基线移除。
5. 记录测试/验收门、已知缺口、依赖和风险；审批完成后将该记录标记为 `Frozen`。

冻结后的规则：

- `Frozen` 基线的范围、验收口径、接口/数据/权限结论不得直接修改；任何调整先创建 `PM-CR-*`。
- 普通源码修复若不改变冻结需求、验收或对外契约，只在所属 Case 留下证据，不另建范围变更单。
- 变更获得批准后，创建新的基线版本并标明 `Supersedes` 与 `Superseded by`；旧基线保持可读。

### 2.2 基线回退

基线回退是治理状态切换，不是 `git reset` 或重写 Linear 历史。适用于批准变更被撤销、实施证明不成立或安全/数据风险要求撤回的场景。

1. 创建新的 `PM-CR-*`，原因选择 `rollback`，指明被撤回的基线和目标基线。
2. 评估并记录需要回退的源码、迁移、配置、数据修复、测试和发布 Case；代码回退必须由其独立实现 Case 执行和验证。
3. 将新基线标为 `Active`，将被撤回版本标为 `Superseded/RolledBack`，但保留全部 SHA、审批和失败证据。
4. 重新执行受影响的回归清单；未恢复到目标基线的 Case 不得关闭，也不得以旧测试结果证明新基线。

## 3. 何时必须提出变更单

满足任一条件即必须先建立 `PM-CR-*`；不允许以评论、口头指示或仅修改源码绕过：

| 触发类别 | 必须提单的具体条件 |
|---|---|
| 范围 | 新增、删除、合并、拆分、重命名 `PM-PRD-*`；改变五类视图、项目/任务生命周期、协作、同步、报表、回收站或 M0–M7 覆盖。 |
| 验收 | 修改成功/失败/边界场景、DoR、DoD、性能阈值、UAT 参与者或验收证据要求。 |
| 契约 | 新增、删除或破坏性修改 API、DTO、事件、同步包、导入导出格式、URL QueryState、错误码或版本语义。 |
| 数据 | 新表/字段/索引/约束、数据迁移、回填、保留策略、备份/恢复、导入冲突策略或不可逆数据操作。 |
| 权限与安全 | PermissionCode、菜单/动作授权、项目角色、ORM Data Filter、租户/应用隔离、文件、审计、PII、安全控制变化。 |
| 架构与依赖 | 聚合边界、事务/Outbox、平台适配、第三方依赖、作业/队列、性能容量或故障恢复策略变化。 |
| 计划 | 关键依赖、里程碑、Case 分解、发布/回滚窗口或责任人变化导致原基线不可实现。 |
| 缺陷/紧急事件 | 已发布安全、数据损坏、越权、不可用或恢复风险需要偏离冻结范围。 |

以下情况不需要范围变更单，但仍须在现有 Case 留证：不改变需求/契约/验收的内部重构、测试补充、文案拼写或构建脚本修复。判断不确定时按“需要变更单”处理。

## 4. 变更单最小字段与影响分析

### 4.1 `PM-CR-*` 模板

在 Linear 中使用以下字段建立记录。可以作为独立子 Case，或在已有实施 Case 的置顶描述中完整填写；两种方式都必须有唯一 `PM-CR-*` 和可访问 URL。

```markdown
## PM-CR-YYYYMMDD-NNN

- Title:
- Status: Draft | InReview | Approved | Rejected | Implementing | Verified | Closed | RolledBack
- Requested by / Date:
- Baseline: PM-BL-... / version ...
- Change type: scope | acceptance | contract | data | permission-security | architecture | schedule | emergency | rollback
- Related requirement IDs: PM-PRD-...
- Related Linear theme / implementation cases: HAO-...
- Reason and observable problem:
- Proposed change and explicit non-goals:
- Alternatives considered and rejection reason:
- Impact analysis: use the mandatory matrix below
- Risk level: Low | Medium | High | Critical
- Required approvers / actual decision / decision time:
- Planned implementation and migration sequence:
- Test, regression, rollout and rollback plan:
- Evidence links: PRD revision, Git SHA/PR, API/UI test, migration/backup, Linear comments
- Baseline result: new PM-BL-... or no new baseline (state reason)
```

### 4.2 强制影响分析矩阵

每个变更单都必须逐项填写 `无影响`、`有影响` 或 `待确认`；`待确认` 不能进入 `Approved`。填写“有影响”时必须提供受影响文件/符号、Case 和验证动作。

| 影响面 | 必填分析内容 |
|---|---|
| 需求与验收 | 受影响 `PM-PRD-*`、PRD 章节、成功/失败/边界场景、DoR/DoD 和 M0–M7 阶段门。 |
| Linear 与计划 | 父主题、子 Case、依赖、里程碑、优先级、责任人、需要新增/取消/重新打开的 Case。 |
| API/DTO/事件 | 路由、请求/响应 DTO、分页/过滤、错误码、幂等键、版本语义、SignalR/WebHook/同步事件和兼容策略。 |
| 数据库与迁移 | 实体、表/字段/索引/约束、数据回填、迁移顺序、可逆性、备份、恢复与 `PRAGMA integrity_check`（适用 SQLite 时）。 |
| 权限与隔离 | PermissionCode、菜单/动作、项目角色、对象级策略、ORM Data Filter、租户/应用/项目边界、审计字段。 |
| 前端与体验 | 路由、QueryKey/缓存、状态管理、Loading/Empty/Error/403/404/409、响应式、键盘、无障碍、未保存确认。 |
| 质量与运行 | 单元/集成/契约/E2E/UAT、真实身份 API/UI、性能容量、并发、作业重启、多实例、监控告警。 |
| 发布与回滚 | Feature flag/配置（如适用）、构建、数据库/数据迁移、灰度、回滚点、不可逆风险、恢复责任人和窗口。 |

## 5. 评审、批准、拒绝与紧急路径

### 5.1 风险分级与批准矩阵

| 风险 | 判定 | 批准前置条件 | 最少批准角色 |
|---|---|---|---|
| Low | 仅澄清未冻结项，无对外契约、数据、权限、验收或计划影响 | 完整变更单、需求和实施负责人复核 | 产品/需求负责人 + 技术负责人 |
| Medium | 影响已排期 Case、验收、内部实现或向后兼容接口 | 完整影响矩阵、回归清单、关联 Case 已更新 | 产品/需求负责人 + 技术负责人 + 测试负责人 |
| High | 影响 API/DTO、数据库迁移、权限/隔离、同步、文件、发布或跨主题依赖 | 迁移/回滚方案、权限/安全分析、真实验证计划 | 产品/需求负责人 + 技术负责人 + 测试负责人 + 安全/数据负责人（按影响面） |
| Critical | 安全事件、数据丢失/损坏、跨租户泄露、服务不可用或不可逆发布风险 | 先止血、最小安全变更、备份/恢复与事后完整复盘 | 技术负责人 + 安全/数据负责人；产品/需求负责人在可用时参与 |

角色是职责，不预设具体人员。项目负责人应在 Linear 记录中将角色映射为实际批准者；缺少必需批准者时，状态必须保持 `InReview`。

### 5.2 正常批准与拒绝

1. 请求人创建 `Draft` 并填满第 4 节所有必填字段。
2. 技术负责人完成影响分析；涉及权限、数据、发布、安全或外部集成时，相关职责人完成专项复核。
3. 测试负责人确认成功、拒绝和关键边界 Case；任何缺口都退回 `Draft`，不得以“开发时再看”批准。
4. 每位批准者在 Linear 留下 `Approved/Rejected`、日期、具体条件和理由。仅变更状态而没有决策文本无效。
5. 批准后才可创建/调整实施 Case、修改冻结文档或开始迁移；实现完成后转 `Verified`，全部证据齐全后 `Closed`。

拒绝路径必须记录：被拒绝的具体提案、理由、可接受替代方案（如有）、对现行基线的结论和是否允许重新提交。`Rejected` 变更不修改基线、不关闭原需求，也不删除审计记录。

### 5.3 紧急路径

仅限 Critical 风险。技术/安全负责人可先批准最小止血措施，但必须：

1. 先记录 `PM-CR-*`、影响的需求/Case、风险和临时措施；无法预先记录时，在恢复服务后首个工作日补齐。
2. 执行前备份或写明为何不可备份；记录执行者、时间、Git SHA、数据/配置变化和恢复点。
3. 在两工作日内补齐完整影响矩阵、产品/测试复审、回归和新的/回退基线决定。
4. 紧急措施不自动成为永久范围；未完成事后批准时必须回退或按普通路径重新提交。

## 6. DoR、DoD 与 M0–M7 阶段门

### 6.1 每个实现 Case 的 Definition of Ready

开始编码前，Case 必须满足：

- 至少关联一个 `PM-PRD-*` 和一个有效 `PM-BL-*`；若是缺口修复，明确原 Case 与失败证据。
- 写明用户可观察的成功、拒绝和至少一个关键边界场景。
- 指定责任人、依赖 Case、目标 M 阶段和验收者；不存在“先做再定义验收”。
- API/DTO、数据、权限、前端、测试和回滚影响已经按第 4.2 节判定；`待确认` 为零。
- 数据/权限/同步/外部集成变更已有迁移、隔离、幂等和失败处理设计。
- 对高风险项，已取得第 5.1 节所需批准。

### 6.2 每个实现 Case 的 Definition of Done

只有同时满足下列项目，Case 才可 `Done`：

- 实现链接到实际 Git SHA；文档、接口、DTO、数据库、权限和前端与批准基线一致。
- 受影响自动化测试通过，新增行为有对应测试；构建使用当前源码和配置。
- 使用有效身份完成真实 API/UI 验收，包含允许、拒绝、租户/应用/项目隔离和关键失败场景；匿名 `401` 不是业务验收。
- 涉及数据库/同步/导入/恢复时，有迁移、备份/回滚或补偿证据；涉及权限时有后端拒绝证据。
- 相关 Linear 主题、子 Case、变更单和追踪矩阵状态已回填；没有未决 `High/Critical` 风险。
- 关闭说明列出命令/测试结果、环境、证据 URL 或 SHA、未覆盖项及其后续 Case。

### 6.3 M0–M7 阶段门

| 阶段 | 进入条件 | 退出/冻结条件 |
|---|---|---|
| M0 需求审计、产品设计、工程基线 | PRD、`PM-PRD-*` 矩阵、架构/权限/数据边界已定位 | `PM-BL-*` 冻结；每项至少映射主题/Case；范围、验收、DoR/DoD、变更治理可追溯。 |
| M1 核心项目/任务域 | M0 冻结；核心实体、状态机、数据权限、迁移和测试计划已批准 | 项目/成员/里程碑/任务/依赖完整链路通过真实身份验收，兼容/回滚清单完成。 |
| M2 多视图与交互 | M1 的任务命令和查询口径稳定 | 五类视图共享同一数据/权限语义；拖动、批量、保存视图、冲突和响应式测试通过。 |
| M3 协作、通知、自动化 | 领域事件和平台适配边界已批准 | 评论/附件/提醒/实时/IM/审批/WebHook 具备幂等、权限回收、失败补偿和端到端证据。 |
| M4 搜索、同步、数据中心、报表 | 数据协议、资源限制和高风险操作策略已批准 | 搜索索引、导入导出、`.bqsync`、备份恢复、审计和报表通过安全/恢复/性能验证。 |
| M5 测试、安全、性能、兼容、迁移 | M1–M4 功能 Case 达到 DoD 或有获批的显式例外 | 自动化、真实身份 E2E、10k 任务性能、跨浏览器、迁移与回滚演练、UAT 记录齐全。 |
| M6 发布、部署、监控、备份、运维 | M5 通过且发布变更单已批准 | 构建/迁移/健康检查/灰度/回滚、监控告警、备份恢复、故障处置和运维手册已实测。 |
| M7 文档、培训、验收、上线支持 | M6 发布候选可用 | 用户/管理员/API/运维文档、培训、验收签字、上线支持、指标复盘和持续反馈机制已留证。 |

阶段门不允许用父主题的状态替代子 Case 证据；任何未通过项应保持在当前阶段或通过已批准的、带到期日的例外记录追踪。

## 7. 审计、证据和关闭记录

### 7.1 必须保留的证据

每个基线和变更单必须可回溯到以下证据，而不是依赖单一聊天记录：

| 证据 | 最小内容 |
|---|---|
| 需求证据 | PRD URL/document ID/修订时间、`PM-PRD-*`、变更前后文本和非目标。 |
| 决策证据 | Linear URL、`PM-CR-*`、审批/拒绝人、时间、条件和理由。 |
| 版本证据 | `PM-BL-*`、版本、四份基线文档的 Git SHA、关联实现 SHA。 |
| 技术证据 | API/DTO/DB/权限/前端/迁移影响矩阵，设计或评审链接。 |
| 质量证据 | 构建、测试、真实身份 API/UI、允许/拒绝、隔离、边界失败、性能/恢复结果。 |
| 发布证据 | 配置/迁移/备份、健康检查、发布/回滚时间、操作者、监控和遗留风险。 |

### 7.2 关闭前检查清单

变更单关闭人逐项确认：

1. `PM-CR-*` 状态为 `Verified`，全部必填字段和影响矩阵已完成。
2. 必要的批准/拒绝/紧急事后审批均有具名决策文本。
3. 受影响 `PM-PRD-*`、追踪矩阵、能力矩阵、架构文档、Linear Case 和 Git SHA 已互相链接；若文档无需变化，已在变更单说明原因。
4. 新基线已冻结，或明确说明该变更未改变冻结口径；存在回退时，目标基线及回归证据已记录。
5. 验收证据符合 6.2 的 DoD；遗留项有独立 Case、负责人、阶段和到期日。
